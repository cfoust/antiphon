# Codex CLI

Движок хуков Codex почти повторяет форму Claude Code, поэтому этот адаптер — почти
дословный порт: тики инструментов, нарратив от модели через MCP, устные итоги с
подстраховкой Stop-хуком и уведомления о блокировке из запросов разрешений.

## Установка

```bash
sh plugins/codex/install.sh
```

Установщик идемпотентно вливается в `~/.codex/` (существующие хуки сохраняются, резервные
копии получают отметку времени): записи хуков в `hooks.json`, блок MCP-сервера в
`config.toml` и скрипты адаптера в `~/.codex/antiphon/`. Затем один раз выполните `/hooks`
внутри Codex, чтобы доверить новые хуки.

## Что вы получаете

- **Старт сессии** вставляет мандат нарратива, и модель по ходу работы вызывает
  `antiphon_task` / `antiphon_progress` / `antiphon_done` / `antiphon_blocked`.
- **Каждый вызов инструмента** задевает ноту аккорда агента (`PostToolUse` →
  `antiphond emit`).
- **Stop** проговаривает последнее сообщение ассистента как устный итог, если модель не
  рассказала свой.
- **Запросы разрешений** запускают цветение ожидания с описанием запроса.

## Особенности

- Codex не передаёт идентификатор сессии MCP-подпроцессам, поэтому одна сессия Codex может
  выглядеть как **две записи в реестре** (канал нарратива и события хуков). Это чистая
  косметика — обе говорят из комнаты, — но знать об этом стоит.
- Обратный канал из комнаты доходит до Codex только через инъекцию в панель терминала
  (tmux).

Всё работает по принципу fail-open: без запущенного Antiphon Codex ведёт себя так, будто
ничего не установлено.
