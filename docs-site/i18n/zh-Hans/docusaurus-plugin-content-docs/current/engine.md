# 引擎

Antiphon 的渲染器是一台专门打造的双耳（binaural）引擎，而不是游戏音频中间件。塑造了整个设计的那条约束是：**一个 Rust DSP 核心，两个宿主，输出逐字节一致。**

## 一个核心，两个目标

引擎（`antiphon-dsp`）是单个 Rust crate：没有 I/O，没有线程，音频路径上没有内存分配。它通过一个 C ABI（`antiphon-ffi`）对外暴露，以两种方式编译：

- 一个 **staticlib**，链接进原生 macOS 应用，由 `AVAudioSourceNode` 拉取；
- 一个 **wasm32 cdylib**，在浏览器里由 `AudioWorklet` 驱动。

一套一致性（parity）校验会把同一场景分别通过两者渲染，并断言输出一致。它们确实一致——误差约在 −155 dBFS，也就是说：网页演示*就是*应用本身的渲染器。

## 信号路径

每个声源、每个 128 帧块、48 kHz 下：

1. **HRTF 卷积**——最小相位 HRIR（128 抽头，SIMD），来自 MIT KEMAR 实测数据集，配合独立的分数延迟 ITD 线路，使双耳时差在你转头时保持精确。
2. **距离**——电平、空气吸收与近场校正。
3. **早期反射**——二阶鞋盒镜像源模型，各表面具有随频率变化的吸声特性；每个声源按能量排序的预算保留最响的镜像。
4. **后期混响**——16 线反馈延迟网络，或分块 BRIR 卷积，按房间预设选择。默认房间是一座以卷积渲染的音乐厅。
5. **头部追踪**——听者姿态旋转整个声场；系数按块渐变，因此运动不会产生咔嗒声。

声源还可以携带辐射指向性（随频率变化的心形指向，正确镜像到反射上）和体积范围（四面体上去相关的卫星抽头）——两者在未启用时都是逐位精确的空操作。

## 头部追踪为何重要

静态双耳音频容易塌缩成"在脑袋里"，对正前方的声源尤其如此。最有力的对策不是个性化 HRTF，而是**运动**：当世界以低延迟随你的头部反向旋转时，大脑就会接受声音的外化。这正是应用用摄像头追踪你头部的原因，也是适配滑块（对 HRTF 做简单的频率缩放）是唯一个性化旋钮的原因：实践中，重活都是追踪干的。

## 离线基准

一切可听的内容都经由 `antiphon-render` 出厂——这是一个离线渲染器，从场景描述生成 WAV 文件。它兼任质量关卡（改动要戴耳机用耳朵评估）和一致性基准。就连营销站上演示影片的配乐也是通过它生成的——宣传片本身就是产品渲染出来的。
