不知从什么时候起，与编程智能体一起工作变成了盯着它们看。如今一次顺利的工作，意味着同时跑着四五个智能体——一个在重构，一个在写文档，一个在追一个时好时坏的测试——而我，像巡查监控屏的保安一样，在终端标签页之间来回切换。智能体一个月比一个月更快、更能干。我这一边的分工却始终没变：轮询、眯眼、滚动。

细想起来，这是一种很奇怪的协作监督方式。没有人靠同时盯着所有人来带团队。你们共享一个空间。你培养出余光般的觉察。你听得出房间里的气氛——谁正进入心流，谁卡住了，谁刚从桌前推开椅子、有话要说。视觉是前台的感官，需要你去指向。听觉才是后台的那一个。它无时无刻不在监听一切，而且分文不取。

于是我造了我想要的那个东西：一个房间。

<Eye />

Antiphon 给每一个智能体会话一个声音，以及你周围虚拟空间里的一个座位。它是一个做真实双耳（binaural）渲染的原生 macOS 应用——实测 HRTF、房间声学，还有一台摄像头注视着你的头部，让世界在你移动时保持原位。智能体不再是一个标签页；它是离你左肩两米远的一个存在。它运行工具时，你听见它的和弦一个音一个音地向下拨响。它沉入一场漫长的构建时，声音降为低沉的三音机器嗡鸣，就像你在工坊另一头听见的车床。它完成时，会开口告诉你——一两句话，用它自己的声音，从它在房间里自己的位置上。

我最在意的交互，是那个零成本的动作：**闭上眼睛。**摄像头察觉到了。桌面淡去，房间浮现，有那么几秒钟，你就站在自己的智能体们中间。转向那个想找你的声音，凝神片刻，它便送上它的汇报。睁开眼，回复框已在等待——键入、回车，你回到自己的工作。用耳朵完成分诊，十秒钟，零标签页。

> 没有提示音，没有闪烁。等待中的智能体只在你的一侧耳畔缓缓积起一片和声——以分钟计——直到你有空理会它。

这种温和正是设计的中心，也是我必须重新学习最多的部分。我用过的每一套通知系统，都把注意力当作可以*攫取*的东西。Antiphon 把它当作你*出借*的东西。这些声音经过几十轮的听感调校才成——更轻、更低、更慢，永远朝那个方向——因为环境工具的失败模式不是过于含蓄，而是惹人厌烦。如果你注意到了声音设计本身，那它就做错了。

## 工程上的赌注

在底层只有一个赌注，并且下得很早：单个 Rust DSP 核心，无 I/O、无线程、音频路径上无内存分配，编译两次——一个 staticlib 放进 Mac 应用，一个 WebAssembly 模块给浏览器。两个构建被验证为逐字节一致，所以[网页演示](https://antiphon.dev/demo.html)不是产品的模型；它*就是*那个渲染器，只不过跑的是脚本化的智能体而非你的。就连首页那部演示影片，也是通过真实引擎离线配乐的。

另一件出乎意料地重要的事：头部追踪胜过 HRTF 个性化。人人都会问起精细的逐耳测量，但真正让大脑相信声音就在外面的是运动——声场随你的头部反向旋转，又快又顺。一个适配滑块加一台摄像头，就能带你走完几乎全程。

一个小小的 Go 守护进程把这一切串起来：智能体插件在 localhost 上接入，会话获得固定的人格与语音（有密钥就用 ElevenLabs 或 OpenAI，没有就用 macOS 内置语音），一切流式传给应用。Claude Code、Codex、OpenCode、Pi 和 Aider 如今都有适配器，协议刻意保持精简——写一个适配器只需一个下午。

## 这个名字

*Antiphon*——ἀντίφωνον——是"应答轮唱"的古老称呼：此起彼应的歌声，隔着空间彼此应答的声音。如今的工作正是这个形状，它配得上比仪表盘更好的东西。整个项目[开源，MIT 许可](https://github.com/cfoust/antiphon)。
