# 引擎

Antiphon 的渲染器是一顆專為此而生的雙耳（binaural）引擎，而不是現成的遊戲音訊中介軟體。塑造一切的設計約束是：**一個 Rust DSP 核心、兩個宿主、逐位元組相同的輸出。**

## 一個核心，兩個目標

引擎（`antiphon-dsp`）是單一 Rust crate：沒有 I/O、沒有執行緒、音訊路徑上不做任何記憶體配置。它透過單一 C ABI（`antiphon-ffi`）對外，並以兩種方式編譯：

- 一個 **staticlib**，連結進原生 macOS 應用程式，由 `AVAudioSourceNode` 拉取；
- 一個 **wasm32 cdylib**，在瀏覽器中由 `AudioWorklet` 驅動。

一套一致性（parity）測試工具會讓同一個場景經由兩邊各渲染一次，並斷言輸出相符。實際上也確實相符——誤差約在 −155 dBFS，也就是說：網頁展示*就是*應用程式的渲染器。

## 訊號路徑

每個聲部、每個 128 影格的區塊、48 kHz：

1. **HRTF 卷積** — 最小相位 HRIR（128 階，SIMD），取自 MIT KEMAR 實測資料集，並搭配獨立的分數延遲 ITD 線路，讓你轉頭時雙耳時間差始終保持精確。
2. **距離** — 音量、空氣吸收與近場修正。
3. **早期反射** — 二階鞋盒鏡像聲源模型，各表面具有隨頻率變化的吸音特性；每個聲源在依能量排序的預算下，只保留最響亮的鏡像。
4. **後期殘響** — 16 線回授延遲網路，或分割式 BRIR 卷積，依房間預設而定。預設的房間是一座以卷積渲染的音樂廳。
5. **頭部追蹤** — 聆聽者的姿態會旋轉整個聲場；係數逐區塊漸變，因此移動時不會出現喀噠聲。

聲部還可以帶有輻射指向性（隨頻率變化的心形指向，正確鏡射到反射上）與體積延展（正四面體上經去相關處理的衛星取樣點）——兩者未啟用時，都是位元級精確的無操作（no-op）。

## 頭部追蹤為何重要

靜態的雙耳音訊往往會塌縮成「就在你頭裡」，對正前方的聲源尤其如此。最有力的解方不是個人化的 HRTF，而是**運動**：當世界以低延遲隨著你的頭反向旋轉，大腦就會接受聲音的外部化。這正是應用程式用網路攝影機追蹤你頭部的原因，也是為什麼貼合滑桿（對 HRTF 做一次簡單的頻率縮放）是唯一的個人化旋鈕：在實務上，粗重的活都是頭部追蹤在扛。

## 離線神諭

所有聽得見的東西，都要經過 `antiphon-render` 才能出貨——它是一個離線渲染器，從場景描述產生 WAV 檔。它既是品質關卡（改動一律戴耳機、用耳朵評估），也是一致性測試的神諭。連行銷網站上示範影片的配樂都是由它產生的——這份簡報，是產品親手渲染出來的。
